## Overview

Before manually installing the cluster scheduling component, you usually need to prepare the corresponding version of the software package and image construction materials. MindCluster components tend to be delivered as "component binary + Dockerfile/YAML + configurationfiles".

This article only summarizes the key points of delivery: **Download range**, **Signature verification**, **Source code entry**.

## Components and Deliverables (Common)

The content varies slightly in different versions/forms, but commonly includes:

- **Ascend Docker Runtime**: Runtime enhancement and mounting capabilities (including installation of run packages, tools and default mounting list)
- **NPU Exporter**: NPU indicator collection (including binary, Dockerfile, DaemonSet YAML)
- **Ascend Device Plugin**: Device registration and scheduling docking (including binary, Dockerfile, YAML adapted to different models/whether Volcano or not)
- **Volcano related** (optional): scheduler component and NPU plug-in library
- **NodeD**: Node status/fault monitoring reporting (including DaemonSet YAML, configuration files, Dockerfile)

## Download and version variables

Official documents often use placeholders to represent version and architecture:

- `{version}`: software version number (example: `7.2.RC1`)
- `{arch}`: CPU architecture (example: `x86_64` / `aarch64`)

It is recommended to record in the delivery note: component version, image tag, runtime type, and K8s version range.

## Software digital signature verification (recommended)

To reduce the risk of tampering during transmission/storage, it is recommended to download the corresponding signature files when downloading the software package, and complete PGP verification before installation/upgrade.

- For verification methods, refer to the official OpenPGP signature verification guide.
- If the verification fails, stop using the package and contact technical support for troubleshooting.

## Open source component source code (optional)

If you need to read the source code or perform secondary development, you can obtain the corresponding component source code from the official repository path (subject to the official list). Common paths include:

- Ascend Docker Runtime
- NPU Exporter
- Ascend Device Plugin
- Volcano（Ascend for Volcano）
- Ascend Operator / NodeD / ClusterD / TaskD
