## Scope of application

The following scenarios usually require deploying the Ascend Device Plugin on the computing nodes:

- Whole card scheduling
- Static/dynamic virtualization scheduling
- Breakpoint continuation training and flexible training
- Inference card failure recovery/rescheduling

If you are only using containerization support or resource monitoring, you can skip it if needed.

## Use constraints (key points for implementation)

- **Driver upgrade sequence**: Before upgrading the driver, it is recommended to stop the business first, and then stop the Device Plugin.
- **Installation sequence with Ascend Docker Runtime**: In most scenarios, it is recommended to **install and restart the runtime** first, and then deploy the Device Plugin; otherwise, you may need to restart/reinstall the Device Plugin to correctly identify it.
- **YAML selection**: Need to match the processor model and whether to use Volcano (different models/scenarios correspond to different YAML).
- **Dynamic library permissions**: It is recommended that the DCMI dynamic library and its parent directory be controlled by root to avoid group/other write permissions.
- **Restart Scenario**: If the basic information of the NPU changes (such as device IP change), the Device Plugin needs to be restarted for re-identification.

## Operation steps (DaemonSet)

### 1) Confirm that the image exists

```bash
docker images | grep k8sdeviceplugin
```

### 2) Select the corresponding YAML

Copy the YAML in the unzipped directory of the software package to the K8s management node, and select the file that matches the model/scenario (for example: training/inference, whether it is Volcano, whether it is a specific development board, etc.).

:::warning
In order to avoid logical anomalies related to "automatically identifying runtime", it is generally not recommended to modify the `metadata.name` of DaemonSet in YAML at will.
:::

### 3) Start

Example (subject to actual YAML):

```bash
kubectl apply -f device-plugin-910-v{version}.yaml
# 或
kubectl apply -f device-plugin-volcano-v{version}.yaml
```

If there are multiple types of Ascend AI processors in the cluster, corresponding YAML needs to be deployed separately.

### 4) Verification

```bash
kubectl get pod -n kube-system | grep -i device-plugin
kubectl describe node <node-name>
```

The presence of `Running` usually indicates that the component started successfully.
