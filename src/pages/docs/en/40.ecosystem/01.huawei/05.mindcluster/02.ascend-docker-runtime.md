## Scope of application

Scenarios that require the following capabilities typically require Ascend Docker Runtime to be installed:

- Containerization support
- Whole card/virtualization scheduling
- Breakpoint continuation training, flexible training
- Inference card failure recovery/rescheduling

If you are only doing resource monitoring, you can usually skip this component.

## Preconditions

- It is recommended to install using root user
- Confirm that user ID `runc` files is 0
- Clarify your runtime scenario: Docker / containerd / K8s integration containerd

## Confirm installation scenario

- Query node:

```bash
kubectl get node
```

- Check the runtime version:

```bash
docker --version
containerd --version
```

- The K8s scenario can confirm the runtime through the node description:

```bash
kubectl describe node <node-name> | grep -i runtime
```

## Installation (Docker scenario)

Execute on all computing nodes:

```bash
cd <path-to-run>
chmod u+x Ascend-docker-runtime_{version}_linux-{arch}.run
./Ascend-docker-runtime_{version}_linux-{arch}.run --check
./Ascend-docker-runtime_{version}_linux-{arch}.run --install
systemctl daemon-reload && systemctl restart docker
```

If you need to specify the installation path (must be an absolute path):

```bash
./Ascend-docker-runtime_{version}_linux-{arch}.run --install --install-path=<abs-path>
```

## Installation (containerd scenario)

```bash
cd <path-to-run>
chmod u+x Ascend-docker-runtime_{version}_linux-{arch}.run
./Ascend-docker-runtime_{version}_linux-{arch}.run --check
./Ascend-docker-runtime_{version}_linux-{arch}.run --install --install-scene=containerd
systemctl daemon-reload && systemctl restart containerd
```

:::warning Ascend Device Plugin will detect the existence of Ascend Docker Runtime at startup. Usually you need to install and restart the container engine first, and then start/restart the Device Plugin.:::

## Frequently Asked Questions

- If your Docker/Containerd configuration files path is not the default location, you may need to specify `--config-file-path` additionally during installation.
- If the containerd installation fails, you often need to check that the runtime configuration in `config.toml` matches the cgroup version (v1/v2), and ensure that the runtime path points to the correct `ascend-docker-runtime`.
