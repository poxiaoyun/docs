## Scope of application

NodeD is typically deployed when using the following capabilities:

- Whole card/virtualization scheduling
- Breakpoint continuation training and flexible training
- Inference card failure recovery/rescheduling

## Operation steps (DaemonSet)

### 1) Confirm that the image exists

```bash
docker images | grep noded
```

### 2) Prepare YAML

Copy the YAML in the decompressed directory of the software package to the K8s management node.

### 3) (Optional) Configure the reporting interval

In scenarios such as breakpoint resume training/elastic training, it may be necessary to adjust the node status reporting interval (example):

```yaml
args: ["/usr/local/bin/noded -logFile=/var/log/mindx-dl/noded/noded.log -logLevel=0 -reportInterval=5"]
```

Note: By default, K8s will set the node to `NotReady` when no node response is received for a period of time; when the APIServer is under heavy pressure, `reportInterval` can be appropriately increased to reduce the pressure (needs to be combined with stability evaluation).

### 4) Start

- When DPC fault detection is not used:

```bash
kubectl apply -f noded-v{version}.yaml
```

- When using DPC fault detection and the environment meets the version requirements:

```bash
kubectl apply -f noded-dpc-v{version}.yaml
```

### 5) Verification

```bash
kubectl get pod -n mindx-dl | grep noded
```

The presence of `Running` usually indicates a successful startup.
