## How to start DCU-Exporter directly in Docker environment?

```bash
docker run --name dcu-exporter-v2 -d --privileged \
  --device=/dev/kfd \
  --device=/dev/mkfd \
  --device=/dev/dri \
  -v /etc/vdev:/etc/vdev \
  -v /etc/hostname:/etc/hostname \
  -v /opt/hyhal:/opt/hyhal \
  -p 16080:16080 image.sourcefind.cn:5000/dcu/admin/base/dcu-exporter:v2.2.0
```

## DCU-Label-Node Failed to add label to DCU node

Solution: Check whether the DCU-Label-Node version is the latest V2.2.0 or above. Versions before V2.2.0 have driver version compatibility issues. For example, V2.1.0 installed on the 6.3.* driver node will cause label addition to fail.

## After applying for DCU resources through DCU-Device-Plugin, the /opt/hyhal directory does not exist in the Pod container

Solution: DCU-Device-Plugin of V2.1.0 and later versions will automatically mount `/opt/hyhal` to the Pod applying for DCU. Versions before V2.1.0 need to be manually mounted in the YAML of applying for DCU.

## The K8s related label in the DCU-Exporter post-deployment indicator is always empty.

Solution:

1. Check whether there is a workload using the DCU device in the k8s cluster
2. Check whether the node `/var/lib/kubelet` is the actual working directory of kubelet
3. Check whether the two environment variables `KUBERNETES_SERVICE_HOST` and `KUBERNETES_SERVICE_PORT` in the DCU-Exporter container are consistent with the actual configuration of the cluster.

## Containerd runs hy-smi in the container and reports an error "Failed to connect service!"

Solution: This is a problem with the 6.3.13 driver, which can be solved by upgrading or downgrading the driver; if upgrade/downgrade is not possible, you can specify `hostNetwork` as a temporary workaround when starting the workload.

## After the 6.2.* driver is upgraded to 6.3.*, hy-smi in the container reports an error "RSMI_NOT_SUPPORTED"

Solution: Execute within the container:

```bash
source /opt/hyhal/env.sh
hy-smi
```

## How to run the application to the specified DCU device in the Docker container

Specify the card number through the environment variable `HIP_VISIBLE_DEVICES`:

```yaml
env:
  - name: HIP_VISIBLE_DEVICES
    value: "0" # 0,1,2,...,n select DCU, -1 for CPU
```

## k8s-vdcu-plugin Pod startup fails after deployment

Troubleshooting suggestions:

1. Check whether the Pod status of the three components `DCU-Device-Plugin`, `vdcu-admission-webhook` and `vdcu-scheduler` is normal; whether the device-plugin node contains the label `dcu=on`
2. If the error message contains: `service \"admission-webhook-dcu-svc\" not found`
   - Check whether `vdcu-admission-webhook` shares the network with the host and whether the host port 443 is occupied
   - Delete legacy resources and redeploy:

```bash
kubectl delete secrets -n kube-system admission-webhook-dcu-certs
kubectl delete mutatingwebhookconfigurations mutating-webhook-dcu-cfg
```

3. V2.1.0 and previous versions also need to check:
   - Whether the `k8s-vdcu-scheduler` version matches the k8s cluster version
   - Whether `k8s-vdcu-scheduler` is scheduled to non-DCU nodes
   - For lower versions of k8s, you may need to change `kubescheduler.config.k8s.io/v1` in the scheduler deployment YAML to `kubescheduler.config.k8s.io/v1beta2`
