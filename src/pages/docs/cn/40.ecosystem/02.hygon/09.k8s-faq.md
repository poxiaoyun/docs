## DCU-Exporter 如何直接在 Docker 环境下启动？

```bash
docker run --name dcu-exporter-v2 -d --privileged \
  --device=/dev/kfd \
  --device=/dev/mkfd \
  --device=/dev/dri \
  -v /etc/vdev:/etc/vdev \
  -v /etc/hostname:/etc/hostname \
  -v /opt/hyhal:/opt/hyhal \
  -p 16080:16080 image.sourcefind.cn:5000/dcu/admin/base/dcu-exporter:v2.2.0
```

## DCU-Label-Node 给 DCU 节点添加标签失败

解决办法：检查 DCU-Label-Node 版本是否为最新 V2.2.0 及以上版本。V2.2.0 之前版本存在驱动版本兼容性问题，例如 V2.1.0 安装在 6.3.* 驱动节点会导致标签添加失败。

## 通过 DCU-Device-Plugin 申请 DCU 资源后，Pod 内容器不存在 /opt/hyhal 目录

解决办法：V2.1.0 及之后版本的 DCU-Device-Plugin 会自动挂载 `/opt/hyhal` 到申请 DCU 的 Pod。V2.1.0 之前版本需要在申请 DCU 的 YAML 中手动挂载。

## DCU-Exporter 部署后指标中 K8s 相关 label 始终为空

解决办法：

1. 检查 k8s 集群中是否存在使用 DCU 设备的工作负载
2. 检查节点 `/var/lib/kubelet` 是否为 kubelet 实际工作目录
3. 检查 DCU-Exporter 容器中 `KUBERNETES_SERVICE_HOST` 与 `KUBERNETES_SERVICE_PORT` 两个环境变量是否与集群实际配置一致

## containerd 容器内运行 hy-smi 报错 "Failed to connect service!"

解决办法：这是 6.3.13 驱动的一个问题，可通过升级或降级驱动解决；若无法升级/降级，可在工作负载启动时指定 `hostNetwork` 作为临时规避方案。

## 6.2.* 驱动升级到 6.3.* 后，容器内 hy-smi 报错 "RSMI_NOT_SUPPORTED"

解决办法：在容器内执行：

```bash
source /opt/hyhal/env.sh
hy-smi
```

## Docker 容器内如何将应用跑到指定的 DCU 设备

通过环境变量 `HIP_VISIBLE_DEVICES` 指定卡号：

```yaml
env:
  - name: HIP_VISIBLE_DEVICES
    value: "0" # 0,1,2,...,n select DCU, -1 for CPU
```

## k8s-vdcu-plugin 部署后 Pod 启动失败

排查建议：

1. 检查 `DCU-Device-Plugin`、`vdcu-admission-webhook`、`vdcu-scheduler` 三个组件 Pod 状态是否正常；device-plugin 节点是否包含标签 `dcu=on`
2. 若报错包含：`service \"admission-webhook-dcu-svc\" not found`
   - 检查 `vdcu-admission-webhook` 是否与宿主机共用网络，宿主机 443 端口是否被占用
   - 删除遗留资源并重新部署：

```bash
kubectl delete secrets -n kube-system admission-webhook-dcu-certs
kubectl delete mutatingwebhookconfigurations mutating-webhook-dcu-cfg
```

3. V2.1.0 及之前版本还需检查：
   - `k8s-vdcu-scheduler` 版本是否与 k8s 集群版本匹配
   - `k8s-vdcu-scheduler` 是否被调度到非 DCU 节点
   - 低版本 k8s 可能需要将 scheduler 部署 YAML 中的 `kubescheduler.config.k8s.io/v1` 改为 `kubescheduler.config.k8s.io/v1beta2`
