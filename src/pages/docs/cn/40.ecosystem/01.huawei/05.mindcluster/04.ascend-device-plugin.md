## 适用范围

以下场景通常需要在计算节点部署 Ascend Device Plugin：

- 整卡调度
- 静态/动态虚拟化调度
- 断点续训、弹性训练
- 推理卡故障恢复/重调度

若仅使用容器化支持或资源监测，可按需跳过。

## 使用约束（落地要点）

- **驱动升级顺序**：升级驱动前建议先停止业务，再停止 Device Plugin。
- **与 Ascend Docker Runtime 的安装顺序**：多数场景建议 **先安装并重启运行时**，再部署 Device Plugin；否则可能需要重启/重装 Device Plugin 才能正确识别。
- **YAML 选择**：需要匹配处理器型号与是否使用 Volcano（不同型号/场景对应不同 YAML）。
- **动态库权限**：DCMI 动态库及其父目录建议由 root 管控，避免 group/other 写权限。
- **重启场景**：若 NPU 基础信息发生变化（例如 device IP 变更），需要重启 Device Plugin 以重新识别。

## 操作步骤（DaemonSet）

### 1）确认镜像存在

```bash
docker images | grep k8sdeviceplugin
```

### 2）选择对应 YAML

将软件包解压目录内的 YAML 拷贝到 K8s 管理节点，并选择适配型号/场景的文件（例如：训练/推理、是否 Volcano、是否特定开发板等）。

:::warning
为避免“自动识别运行时”相关逻辑异常，通常不建议随意修改 YAML 中 DaemonSet 的 `metadata.name`。
:::

### 3）启动

示例（以实际 YAML 为准）：

```bash
kubectl apply -f device-plugin-910-v{version}.yaml
# 或
kubectl apply -f device-plugin-volcano-v{version}.yaml
```

若集群存在多种类型的昇腾 AI 处理器，需要分别部署对应 YAML。

### 4）验证

```bash
kubectl get pod -n kube-system | grep -i device-plugin
kubectl describe node <node-name>
```

出现 `Running` 通常表示组件启动成功。
