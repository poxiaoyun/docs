## 适用范围

使用资源监测时通常需要部署 NPU Exporter，并将指标对接到 Prometheus 或 Telegraf。

## 使用约束（落地要点）

- **驱动升级顺序**：Exporter 会周期性调用驱动接口；升级驱动前建议先停止业务，再停止 Exporter。
- **K8s 版本**：K8s 1.24+ 场景可能需要额外准备 `cri-dockerd`（以你的集群运行时与部署方式为准）。
- **动态库权限**：DCMI 动态库及其父目录需要满足更严格的属主与写权限约束（建议由 root 管理，避免 group/other 写权限）。
- **容器部署风险**：容器化运行往往需要特权与 socket 挂载，安全要求高时可考虑二进制部署。

## 部署方式一：容器化（DaemonSet，推荐）

### 1）确认镜像存在

在计算节点检查镜像（示例）：

```bash
docker images | grep npu-exporter
```

或 containerd：

```bash
ctr -n k8s.io c ls | grep npu-exporter
```

### 2）选择并修改 YAML

将软件包解压目录内的 YAML 拷贝到 K8s 管理节点。

- Docker / containerd 场景差异主要体现在启动参数 `-containerMode` 以及运行时 socket 的挂载路径。
- 如果需要调整端口/更新周期/日志等，可修改启动参数（如 `-port`、`-updateTime`、`-logFile` 等）。

### 3）启动与检查

```bash
kubectl apply -f npu-exporter-v{version}.yaml
kubectl get pod -n npu-exporter
```

若出现 `CrashLoopBackOff`，优先检查日志目录权限、DCMI 动态库权限与 socket 挂载路径是否正确。

## 部署方式二：二进制（安全要求高时）

思路是将 `npu-exporter` 放到固定目录，通过 systemd service/timer 延迟启动（避免设备未就绪）。

- 建议为日志目录配置正确的属主/权限
- 如需获取容器相关信息，可能需要为二进制授予能力（capability），并在变更后重启服务
